## 2026-03-22 开发日志

### 一、AOF 同步策略优化

#### 1. 背景
原有 `Append()` 方法每次写入都执行 `Flush()` + `Sync()`，磁盘 I/O 成为性能瓶颈。需要引入类似 Redis 的 AOF 策略机制，平衡数据安全与性能。

#### 2. 三种同步策略

| 策略 | 说明 | 数据安全 | 性能 |
|------|------|----------|------|
| `always` | 每次写入都 fsync | 最高 | 最低 |
| `everysec` | 每秒 fsync 一次 | 高（最多丢1秒）| 高 |
| `no` | 由 OS 决定 | 低 | 最高 |

#### 3. 新增文件

**`internal/persist/sync.go`** - 同步策略管理模块

```go
// SyncPolicy AOF 同步策略
type SyncPolicy int

const (
    SyncAlways   SyncPolicy = iota // 每次写入都 fsync
    SyncEverySec                     // 每秒 fsync 一次（默认）
    SyncNo                           // 不主动 fsync，依赖 OS
)

// Syncer 管理 AOF 同步策略
type Syncer struct {
    mu         sync.RWMutex
    policy     SyncPolicy
    needSync   bool            // everysec 模式用
    ticker     *time.Ticker    // 定时器
    stopChan   chan struct{}
    syncFunc   func() error    // sync 回调函数
    running    bool
}
```

核心方法：
- `AfterWrite()` - 根据策略决定是否立即 sync
- `NeedFlush()` - 判断是否 flush 到 OS 缓冲区
- `Start()/Stop()` - 启停后台同步协程

#### 4. 修改文件

**`internal/persist/aof.go`**:
- 添加 `syncer *Syncer` 字段
- `Append()` 根据策略执行不同逻辑
- 添加 `SetSyncPolicy()` / `GetSyncPolicy()` 方法

**优化后的 Append 逻辑**:
```go
func (p *Persistence) Append(command string) error {
    p.mutex.Lock()
    
    // 写入命令
    p.writer.WriteString(command + "\n")
    
    // 根据策略决定是否 flush
    if p.syncer.NeedFlush() {
        p.writer.Flush()
    }
    
    // 根据策略决定是否立即 sync
    if p.syncer.AfterWrite() {
        p.file.Sync()  // 仅 always 模式执行
    }
    
    p.mutex.Unlock()
}
```

### 四、技术讨论

#### select { case ...; default: } 的用法

在 `Stop()` 方法中使用非阻塞发送模式：

```go
select {
case s.stopChan <- struct{}{}:  // 尝试发送
default:                         // 如果被阻塞，直接返回
}
```

**目的**: 防止 `Stop()` 被重复调用时阻塞程序
- 第一次调用：信号被接收，后台协程退出
- 第二次调用：无接收者，走 default 分支直接返回

这是一种**防御性编程**技巧，确保方法可被安全地多次调用。
