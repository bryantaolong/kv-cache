### 2. AOF 持久化 ✅

**实现文件**: `internal/persistence/aof.go`

实现了 Append-Only File 持久化机制：

**工作原理**:
1. **写入**：所有写命令自动追加到 `data/appendonly.aof`
2. **同步**：每次写入后强制 fsync，确保数据安全
3. **加载**：启动时自动重放 AOF 文件恢复数据
4. **优雅关闭**：捕获 SIGINT/SIGTERM 信号，确保数据保存

**使用方式**:
```bash
# 默认启用持久化
$ go run main.go

# 指定数据目录
$ go run main.go -data /path/to/data

# 禁用持久化（内存模式）
$ go run main.go -no-persist
```

**验证结果**:
```
# 第一次运行创建数据
$ echo "SET name alice" | go run main.go
# 生成 data/appendonly.aof

# 第二次运行自动加载
$ go run main.go
* Loaded 12 commands from AOF
kv-cache> GET name
"alice"
```

### 3. 项目结构重组 ✅

**原结构问题**:
- 命令层与存储层耦合在 `pkg/store/`
- CLI 包含持久化逻辑，职责不清
- 多个空目录（`pkg/cmd/`, `pkg/resp/` 等）
- `main.go` 过于臃肿（400+ 行）

**新结构**（标准 Go 项目布局）：

```
kv-cache/
├── cmd/kv-cache/           # 应用程序入口
│   └── main.go            # 60行简洁入口，只负责初始化
├── internal/              # 内部实现（不对外暴露）
│   ├── cli/               # 命令行交互层
│   │   └── cli.go         # TUI 实现、命令解析
│   ├── persistence/       # 持久化层
│   │   └── aof.go         # AOF 实现
│   └── storage/           # 存储层
│       ├── store.go       # MemoryStore 引擎
│       ├── types.go       # 基础类型定义
│       ├── cmd_*.go       # 各类型命令实现
│       └── *_test.go      # 单元测试
├── pkg/                   # 对外暴露的库（预留）
│   └── client/            # 未来：Go 客户端 SDK
├── data/                  # 数据目录（运行时生成）
│   └── appendonly.aof     # AOF 持久化文件
└── docs/                  # 文档
```

**改进点**:
1. **职责分离**：CLI、Persistence、Storage 三层独立
2. **标准布局**：采用 `cmd/` + `internal/` 标准 Go 项目结构
3. **main.go 解耦**：从 400+ 行精简到 60 行
4. **扁平化存储层**：移除 `types/` 子目录，避免包访问问题

**目录职责**:
| 目录 | 职责 | 可见性 |
|------|------|--------|
| `cmd/` | 应用程序入口 | 公共 |
| `internal/cli/` | 命令行交互 | 内部 |
| `internal/persistence/` | AOF 持久化 | 内部 |
| `internal/storage/` | 存储引擎 + 命令 | 内部 |
| `pkg/` | 对外库（预留） | 公共 |

## 关键设计决策

### 1. AOF 持久化策略
- **同步策略**：每次写命令立即 fsync（最安全，性能可接受）
- **加载机制**：启动时逐行重放，不阻塞主流程
- **错误处理**：加载时跳过错误命令，打印警告继续

### 2. 命令行架构
```
main.go → CLI → Command → Storage
             ↓
         Persistence (AOF)
```
- CLI 解析输入，调用 Storage 方法
- 写命令自动通知 Persistence 记录
- 完全解耦，便于后续扩展（如添加 RESP 服务器）

### 3. 项目布局选择
采用标准 Go 项目布局而非过度设计：
- ✅ `cmd/`：可执行程序入口
- ✅ `internal/`：不对外暴露的核心实现
- ⏳ `pkg/`：预留对外库（如客户端 SDK）
- ❌ 不使用 `pkg/api/`, `pkg/utils/` 等过度分层

## 遇到的问题与解决

### 1. 子目录包访问问题
**问题**：`internal/storage/types/` 下的类型在父目录不可见  
**解决**：扁平化，将类型文件移至 `internal/storage/`  
**教训**：Go 中子目录是不同的包，同包文件应放同一目录

### 2. Import 路径循环依赖
**问题**：移动文件后大量 import 路径失效  
**解决**：批量替换 `"kv-cache/pkg/store"` → `"kv-cache/internal/storage"`  
**工具**：使用 `grep` + `sed` 批量替换

### 3. CLI 与 Persistence 耦合
**问题**：最初将 AOF 放在 `pkg/cli/persist.go`  
**解决**：提取到独立的 `internal/persistence/` 包  
**好处**：职责清晰，便于后续添加 RDB 等其他持久化方式
